# Pitfalls Research

**Domain:** YouTube Shorts engagement upgrade — Ken Burns, scene transitions, LLM hooks, TTS prosody
**Researched:** 2026-03-30
**Confidence:** HIGH (Remotion and edge-tts from official docs; LLM/hook patterns from multiple corroborating sources)

---

## Critical Pitfalls

### Pitfall 1: TransitionSeries Silently Shortens Total Video Duration

**What goes wrong:**
Switching from plain `<Sequence>` to `<TransitionSeries>` for scene cuts causes the total `durationInFrames` to shrink. A 10-scene video with 30-frame transitions loses 9 × 30 = 270 frames (~9 seconds at 30fps). The audio continues past the end of the video, or the video ends early and gets clipped by the Remotion renderer.

**Why it happens:**
`<TransitionSeries>` is designed to overlap adjacent sequences during the transition period, reducing total duration by design. The formula is: `total = sum(all sequence durations) - sum(all transition durations)`. Developers port over existing per-image frame budgets without accounting for this shrinkage.

**How to avoid:**
- Keep `durationInFrames` on `<Composition>` (or `calculateMetadata`) as the audio-derived truth — compute it from `durationInSeconds * fps` as is already done in `Root.tsx`.
- Do NOT expand per-image sequence durations to compensate; instead, account for transition overlap when distributing frames: `framesPerImage = Math.floor((durationInFrames + (n-1)*TRANSITION_FRAMES) / n)`.
- After adding transitions, add an assertion in `render.mjs` or a dev-time check that computes the expected TransitionSeries total and compares it against `durationInSeconds * fps`.

**Warning signs:**
- Video ends before audio finishes (audio cut off or silent tail).
- Subtitle timestamps go out of sync past the midpoint — subtitles appear to lag.
- `calculateMetadata` in `Root.tsx` returns a shorter `durationInFrames` than `durationInSeconds * 30`.

**Phase to address:** Ken Burns / transitions phase — before any scene transition code ships.

### Pitfall 2: Ken Burns Pan Exceeds Image Bounds, Exposing Black Edges

**What goes wrong:**
Combining `translateX`/`translateY` pan with `scale` causes the image to drift far enough that a black or white border appears at one edge of the frame. This is especially visible on AI-generated images that already touch the composition edges (1080×1920 portrait).

**Why it happens:**
The current implementation in `VideoShort.tsx` scales from `transformOrigin: "center center"` with no pan — only zoom-in or zoom-out. When pan is added (to create the full Ken Burns drift), the translation is applied after the scale, so a 4% scale-up only provides ~22px of edge buffer at 1080px wide. Any `translateX` greater than ~20px exposes the uncovered background.

**How to avoid:**
- Keep pan magnitude proportional to scale headroom: if scale tops out at 1.04 (4% zoom), max `translateX` = `(width * (scale - 1)) / 2 = 1080 * 0.02 = ~21px`. Scale to 1.08–1.10 if you want noticeable drift.
- Apply pan as a percentage of the image dimensions, not as a fixed pixel value.
- Use `overflow: hidden` on the wrapping `<AbsoluteFill>` — this is the safety net when math is slightly off.
- Test with images that have important content near edges (logos, text generated by Gemini).

**Warning signs:**
- Thin black stripe appears on one side of the image during motion, especially at the start or end of a clip.
- Motion feels "jumpy" at the turn of a zoom-out — the image snaps to reveal background.

**Phase to address:** Ken Burns implementation phase.

### Pitfall 3: edge-tts Pitch Parameter Has No Effect in Current Versions

**What goes wrong:**
Adding `pitch="+50Hz"` to `edge_tts.Communicate()` produces audio identical to the pitch-unmodified call. The video ships without the intended vocal energy variation, yet the code appears to be wired correctly.

**Why it happens:**
The `--pitch` CLI option and corresponding Python API parameter were removed in edge-tts v6.0.3 because Microsoft's server-side service no longer honours pitch changes. The parameter is silently ignored rather than raising an error — making this failure invisible.

**How to avoid:**
- Only rely on `rate` for prosody improvement. `rate="+20%"` to `"+35%"` is the single reliable lever for a more energetic feel.
- Do not expose `pitch` as a config option — it will create false expectations.
- If pitch variation is truly required, that is the trigger to switch to ElevenLabs or Azure TTS (out of scope for this milestone, but noted for future roadmap).
- Pin `edge-tts` version in `requirements.txt` and document that pitch was removed in v6.0.3.

**Warning signs:**
- Generated audio sounds identical regardless of the pitch value in config.
- `edge-tts --pitch=+100Hz "test"` produces the same output as `edge-tts "test"`.

**Phase to address:** TTS prosody phase.

### Pitfall 4: LLM Hook Returns Formatting Marks or Non-Plain Text

**What goes wrong:**
Ollama generates a hook sentence wrapped in quotes, prefixed with "Hook:" or a bullet marker, or surrounded by markdown bold (`**bold**`). This raw output gets prepended to the script and passed to edge-tts, which reads the punctuation aloud or produces odd pauses.

**Why it happens:**
The existing `generate_script()` already strips `*` characters with regex, but there is no equivalent stripping for the hook. Local Ollama models (especially smaller ones like Mistral 7B or Llama 3.1 8B) are not as instruction-following as hosted models and frequently add formatting despite `ONLY RETURN` instructions.

**How to avoid:**
- Apply the same `re.sub(r"\*", "", hook)` strip that script generation uses.
- Additionally strip: leading/trailing quotes (`strip('"\')`), lines starting with "Hook:", "Opening:", bullet characters (`-`, `•`), and markdown headers (`#`).
- Constrain the Ollama prompt to produce exactly one sentence with a hard length limit (under 15 words recommended for Shorts hooks).
- Add a validation step: if the hook contains a newline, retry once and fall back to a hardcoded template hook if the second attempt also fails.

**Warning signs:**
- TTS audio contains a spoken asterisk sound or "asterisk" word.
- The hook segment of the rendered video subtitle shows markdown syntax literally.
- Generated audio for the hook is significantly longer than the first scene's frame budget.

**Phase to address:** Hook generation phase.

### Pitfall 5: Hook Semantically Disconnected from Script Body

**What goes wrong:**
The LLM generates an attention-grabbing hook (e.g., "Did you know the universe is DYING?") but the script body starts with a different angle (e.g., facts about black holes). The viewer is baited into watching but the content does not deliver the promise — causing a sharp drop in the retention curve after the first 5 seconds, which YouTube's algorithm counts against the video.

**Why it happens:**
If hook generation is a separate LLM call that only receives the `topic`, it has no knowledge of the script's specific content direction. The hook and body are semantically misaligned.

**How to avoid:**
- Generate the hook from the already-generated `script` text, not just the `topic`. Pass the full script to the hook prompt so the LLM can extract a genuine teaser from the actual content.
- Alternatively, generate hook-then-script in a single prompt, using a structured output format: `{ "hook": "...", "body": "..." }`.
- The hook must make a claim that the script body actually backs up.

**Warning signs:**
- When reading hook + script text together, the hook question is never answered in the body.
- The hook uses superlatives ("shocking", "world-changing") but the script is factual and mild in tone.

**Phase to address:** Hook generation phase.

### Pitfall 6: Ken Burns `interpolate` Without Clamping Causes Scale Runaway

**What goes wrong:**
The `interpolate(localFrame, [0, framesPerImage], ...)` call in `ImageSlideshow` uses `extrapolateLeft: "clamp"` and `extrapolateRight: "clamp"` — but this protection is only present because it was written correctly. If this guard is removed or a new animation variable is added without clamping, the scale value continues growing past the intended range. At frame 60 past the end of a segment, a scale that was meant to reach 1.04 reaches 1.08, causing extreme zoom that makes the image unrecognisable.

**Why it happens:**
Remotion's `interpolate()` defaults to `extrapolateRight: "extend"` — the most dangerous default for video. Developers often copy a snippet that shows the input/output ranges but omit the `extrapolateLeft/Right: "clamp"` options when extending the animation.

**How to avoid:**
- All `interpolate()` calls for scale, translate, or opacity in this codebase must include both `extrapolateLeft: "clamp"` and `extrapolateRight: "clamp"` — treat this as a project-wide convention.
- The existing code in `VideoShort.tsx` already follows this convention; any new animation code must do the same.
- Add a comment above each `interpolate()` call explaining why clamp is required.

**Warning signs:**
- Studio preview shows images that balloon in size or pan off-screen during the last frames of a segment.
- Scale values printed to console during a debug render exceed the intended maximum.

**Phase to address:** Ken Burns implementation phase.

### Pitfall 7: `calculateMetadata` / `durationInSeconds` Float Precision Causes 1-Frame Gaps

**What goes wrong:**
Audio files rarely have durations that are exact multiples of `1/fps`. `Math.ceil(durationInSeconds * 30)` produces the correct frame count, but if `durationInSeconds` is passed through JSON serialisation (Python → `props.json` → Node.js), floating-point rounding can cause the computed frame count to differ by ±1 frame. This creates a 1-frame freeze or black flash at the end of the video.

**Why it happens:**
Python's `float` → JSON → JavaScript `Number` round-trip is not lossless for values like `32.033333...`. The `durationInSeconds` value written to the props JSON by `combine_remotion()` may lose the last few digits.

**How to avoid:**
- In Python, round `durationInSeconds` to 3 decimal places before writing to the props JSON: `round(duration, 3)`.
- In `Root.tsx`, `Math.ceil` already provides a floor guarantee, but add 0.1s of padding to `durationInSeconds` so the last audio frame is never cut off: `durationInSeconds + 0.1`.
- Verify by inspecting the final rendered MP4 duration in `ffprobe` output against the expected audio duration.

**Warning signs:**
- Rendered video is 1–2 frames shorter than the audio file.
- Final frame freezes or shows black for a brief flash when scrubbing to end of video in Studio.

**Phase to address:** Ken Burns / transitions phase (touches `durationInSeconds` handling).

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| Hard-code `CROSSFADE_FRAMES = 15` as a constant | Quick to write | All transitions must be the same length; can't tune per-category preset | MVP only — add to `CategoryPreset` interface when presets expand |
| Generate hook in a separate Ollama call (not combined with script) | Simpler code | Two round-trips to Ollama; hook can diverge from script semantics | Never — pass script to hook prompt from day one |
| Skip `--rate` tuning for non-English voices | Fast English-only launch | Rate that sounds energetic in en-US sounds robotic in other locales | Acceptable until Thai/other-language clips are in scope |
| Use fixed `transformOrigin: "center center"` for all Ken Burns | Correct for zoom-only | Cannot do corner-anchored pans (zoom into top-left, pan to bottom-right) | Acceptable for this milestone |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| edge-tts `Communicate` | Passing `pitch` parameter expecting it to change output | Only pass `rate` (e.g., `rate="+20%"`); pitch is silently ignored since v6.0.3 |
| edge-tts `Communicate` | Passing rate as a bare number (`rate=20`) | Must be a string with sign and percent: `rate="+20%"` |
| Remotion `TransitionSeries` | Using it in the same file as a plain `<Sequence>` without installing `@remotion/transitions` | Install `@remotion/transitions` package; it is separate from core `remotion` |
| Remotion `staticFile()` | Referencing asset paths that have not been staged to `remotion/public/assets/` before render | Python's `combine_remotion()` must stage all files before spawning the Node subprocess |
| Ollama hook prompt | Not setting `format="json"` when expecting structured output | Local models hallucinate extra text; use `format="json"` or apply post-processing regex unconditionally |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| High `CROSSFADE_FRAMES` value (>30) with many images | Render time increases non-linearly; both scenes rendered simultaneously for longer overlap | Keep crossfade ≤ 15–20 frames; more images = shorter viable crossfade | Every image pair that overlaps adds 2× rendering work for that duration |
| Ollama retry loop without back-off for hook generation | LLM server gets hammered on malformed output; Python process stalls for 30–60s | Limit retries to 2; fall back to template hook after 2 failures | When local Ollama is loaded from cron and another job is running |
| `rem_temp_files()` deleting staged assets mid-render | Remotion render fails with "file not found" | `rem_temp_files()` only runs at the start of a run, before staging; verify call order | Only on concurrent cron runs (unlikely, but possible) |

## "Looks Done But Isn't" Checklist

- [ ] **Ken Burns**: Verify effect is visible in Studio preview — a 4% scale change is imperceptible on some monitors at low zoom; confirm at 1:1 pixel zoom.
- [ ] **Ken Burns**: Confirm no black edges appear by testing with an image that has a high-contrast element near each of the four sides.
- [ ] **Transitions**: Confirm total `durationInFrames` of the TransitionSeries matches `durationInSeconds * 30` after subtracting transition overlap; log both values at render time.
- [ ] **Hook**: Confirm hook sentence ends with proper punctuation (`.`, `?`, `!`) so Whisper segments it correctly and TTS does not run it into the next sentence without pause.
- [ ] **TTS prosody**: Play back the audio at 1.5× speed in a media player to sanity-check — the `+20%` rate increase on top of a fast playback will reveal any robotic artefacts.
- [ ] **Subtitle sync**: After adding transitions, re-check that subtitle timestamps (from Whisper SRT) still align with spoken words in the final MP4 — the SRT is generated from audio, not video, so it is immune to frame-count changes, but verify visually.
- [ ] **Props JSON**: Confirm `durationInSeconds` written to props JSON is the actual `ffprobe`-measured audio duration, not the `script` length estimate.

## Recovery Strategies

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| TransitionSeries shortens video, audio clipped | MEDIUM | Add transition overlap to per-image frame budget formula; re-render one test video; no Python changes needed |
| Ken Burns black edges | LOW | Increase scale range from 1.0–1.04 to 1.0–1.08 in `VideoShort.tsx`; re-render |
| Pitch ignored, no vocal variation | LOW | Remove pitch param; increase rate; document limitation |
| Hook output contains markdown/formatting | LOW | Add stripping regex to hook post-processing; 30-minute fix |
| Hook disconnected from script | MEDIUM | Refactor hook prompt to receive `script` not just `topic`; may need Ollama call reorder |
| `interpolate` scale runaway | LOW | Add missing `extrapolateRight: "clamp"` to new animation call |
