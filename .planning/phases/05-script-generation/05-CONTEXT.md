# Phase 5: Script Generation - Context

**Gathered:** 2026-04-01
**Status:** Ready for planning

<domain>
## Phase Boundary

Implement `Podcast.generate_script()` — takes a topic, calls Ollama 3 times (intro+act1 / act2 / act3+outro) with running summary injection between calls, returns a flat list of 14 scene dicts, and persists the result to `.mp/podcast_<id>/script.json`. No image generation, no TTS, no rendering — pure script output.

</domain>

<decisions>
## Implementation Decisions

### Scene Data Structure

- **D-01:** Script output is a **plain list of dicts** — `[{"narration": "...", "image_prompt": "..."}]` — 14 items total. No Pydantic models. Phase 6 consumes the list directly.

### Act Generation Loop

- **D-02:** Generate in **3 LLM calls**:
  1. Call 1 — intro(1) + act1(4) → 5 scenes
  2. Call 2 — act2(4) → 4 scenes, with summary of Call 1 injected into prompt
  3. Call 3 — act3(4) + outro(1) → 5 scenes, with summary of Calls 1+2 injected into prompt
- **D-03:** Running summary is a brief plain-text recap of prior scenes/acts, generated by Claude after each call and injected as context into the next prompt — keeps narrative coherent without exceeding Ollama context limits (per SCRP-02).

### Narrator Persona Injection

- **D-04:** Persona is injected via **system prompt** on every `generate_text_structured()` call. System prompt sets role/tone: `"You are {name}, {persona}. Narrate in a dark comic noir style..."` — consistent voice across all 3 calls.

### Storage & Caching

- **D-05:** Completed script is saved to **`.mp/podcast_<id>/script.json`** — flat JSON array of 14 scene dicts. Consistent with existing `.mp/` scratch directory pattern. Phase 6 reads this file to begin asset generation.
- **D-06:** `podcast_<id>` directory is created by `generate_script()` before writing — use topic slug + timestamp as ID (e.g. `podcast_black-holes_20260401`).

### Claude's Discretion

- Summary generation approach (LLM-generated vs rule-based extract) — Claude decides based on simplicity
- Exact prompt wording for scene/act instructions — Claude designs prompts
- JSON schema passed to `generate_text_structured()` for structured scene output — Claude defines
- Error handling / retry on malformed JSON output from Ollama — Claude decides retry strategy

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `llm_provider.generate_text_structured(prompt, system_prompt, schema)` — structured JSON output via Ollama; use for all 3 act generation calls
- `llm_provider.generate_text(prompt)` — plain text generation; use for running summary generation between calls
- `get_podcast_narrator()` → dict with `name`, `persona`, `tts_voice`, `tts_rate` — build system prompt from these
- `get_podcast_style_prompt()` → string — prepend to every `image_prompt` in scene output

### Established Patterns
- LLM calls go through `llm_provider` functions only — no direct Ollama client in class files
- Config access: call getter each time (no caching), re-reads `config.json` on every call
- Temp/scratch files go in `.mp/` at project root — subdirectory per episode

### Integration Points
- `src/classes/Podcast.py` `generate_script()` → fills in the stub, saves to `.mp/podcast_<id>/script.json`, returns list of 14 dicts
- `generate_assets()` (Phase 6) will read `.mp/podcast_<id>/script.json` to get scenes

</code_context>

<specifics>
## Specific Ideas

- Episode ID format: `podcast_{topic_slug}_{YYYYMMDD}` — deterministic enough to resume, unique enough to avoid collisions
- Running summary between acts should be concise (3-5 sentences) — enough for narrative continuity, not so long it wastes context
