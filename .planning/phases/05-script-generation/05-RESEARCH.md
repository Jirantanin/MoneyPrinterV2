## User Constraints (from CONTEXT.md)

### Locked Decisions

- **D-01:** Script output is a **plain list of dicts** — `[{"narration": "...", "image_prompt": "..."}]` — 14 items total. No Pydantic models in the output. Phase 6 consumes the list directly.
- **D-02:** Generate in **3 LLM calls**:
  1. Call 1 — intro(1) + act1(4) → 5 scenes
  2. Call 2 — act2(4) → 4 scenes, with summary of Call 1 injected into prompt
  3. Call 3 — act3(4) + outro(1) → 5 scenes, with summary of Calls 1+2 injected into prompt
- **D-03:** Running summary is a brief plain-text recap of prior scenes/acts, generated by the LLM after each call and injected as context into the next prompt.
- **D-04:** Persona is injected via **system prompt** on every `generate_text_structured()` call. System prompt sets role/tone from narrator config.
- **D-05:** Completed script saved to **`.mp/podcast_<id>/script.json`** — flat JSON array of 14 scene dicts.
- **D-06:** `podcast_<id>` directory created by `generate_script()` before writing — topic slug + timestamp as ID (e.g. `podcast_black-holes_20260401`).

## Summary

Phase 5 implements `Podcast.generate_script(topic)` — a pure LLM orchestration method that calls Ollama three times with progressive context injection, producing a flat 14-scene list and persisting it to disk. The method signature must accept `topic: str` (the stub currently takes no arguments — the signature change is required by SCRP-01).

The technical core is well-understood: `generate_text_structured()` already exists and works correctly. The JSON schema for each call must specify an array of scene dicts with exact `narration` and `image_prompt` fields. Between calls, `generate_text()` produces a 3-5 sentence summary that is prepended to the next call's user prompt. The narrator system prompt is constructed once from `get_podcast_narrator()` and reused across all three calls.

The one non-obvious risk is Qwen3:8b's thinking mode. The installed Ollama SDK (0.6.1) cleanly separates thinking tokens into `message.thinking` (not `message.content`), so JSON output in `content` is uncontaminated by `<think>` tags when `format=schema` is used. No special handling is needed.

**Primary recommendation:** Implement `generate_script(topic: str)` with three `generate_text_structured()` calls using a `scenes` list schema, one `generate_text()` summary call between each pair, narrator system prompt reused across all calls, and directory + JSON write at the end.

## Architecture Patterns

### Recommended Project Structure

No new files or directories beyond the phase scope. All implementation lives in:

```
src/
└── classes/
    └── Podcast.py       # generate_script() implementation (stub → live)
```

Episode scratch data:
```
.mp/
└── podcast_{slug}_{YYYYMMDD}/
    └── script.json      # flat JSON array of 14 scene dicts
```

### Pattern 1: Signature Change — topic Parameter

The current stub `generate_script(self) -> None` must become `generate_script(self, topic: str) -> list`. The return type changes to `list` (14 dicts). The caller in `run()` must also pass `topic`.

**What:** `run()` already calls `self.generate_script()` — it will need to receive topic from `__init__` or be refactored to accept topic at call time. Simplest approach: store topic on `self` in `__init__` (or accept as `run(topic)`) and pass through.

**Established pattern in YouTube.py:** Topic is stored as `self.subject` and passed to all sub-methods implicitly via instance state. Use `self.topic` for Podcast.

### Pattern 2: JSON Schema for Structured Scene Output

**What:** The `format=` parameter on `generate_text_structured()` accepts a JSON Schema dict. Use a minimal inline dict — no Pydantic (D-01).

```python
# Source: YouTube.py generate_hook() + ollama SDK 0.6.1 _types.py
SCENE_SCHEMA = {
    "type": "object",
    "properties": {
        "scenes": {
            "type": "array",
            "items": {
                "type": "object",
                "properties": {
                    "narration": {"type": "string"},
                    "image_prompt": {"type": "string"},
                },
                "required": ["narration", "image_prompt"],
            },
        }
    },
    "required": ["scenes"],
}
```

Parse with: `json.loads(raw_json)["scenes"]` — returns a list of `{"narration": ..., "image_prompt": ...}` dicts.

### Pattern 3: Narrator System Prompt Construction

**What:** Build system prompt once at the top of `generate_script()`, reuse for all 3 calls.

```python
# Source: CONTEXT.md D-04, config.get_podcast_narrator()
narrator = get_podcast_narrator()  # {"name": ..., "persona": ..., "tts_voice": ..., "tts_rate": ...}
system_prompt = (
    f"You are {narrator['name']}, {narrator['persona']}. "
    "Narrate in a compelling storytelling voice. "
    "Output ONLY valid JSON matching the provided schema. "
    "No markdown, no asterisks, no extra commentary."
)
```

This pattern mirrors YouTube.py's hook system prompt exactly.

### Pattern 4: Inter-Call Summary via generate_text()

**What:** After receiving scenes from Call 1 (and Call 2), generate a 3-5 sentence plain-text summary of what was narrated, to inject into the next call's user prompt.

```python
# Source: CONTEXT.md D-03, D-02
summary_prompt = (
    f"Summarize the following podcast script scenes in 3-5 sentences, "
    f"capturing the key narrative points covered so far:\n\n"
    + "\n".join(s["narration"] for s in scenes_so_far)
)
running_summary = generate_text(summary_prompt)
```

Then inject: prepend `f"Story so far:\n{running_summary}\n\n"` to the user prompt of the next act call.

### Pattern 5: Episode ID and Directory Creation

**What:** Create `.mp/podcast_{slug}_{YYYYMMDD}/` before writing. Slug derived from topic.

```python
# Source: CONTEXT.md D-06 specifics
```

### Pattern 6: style_prompt Prepend on image_prompt

**What:** After collecting all 14 scenes, prepend `self.style_prompt` to each scene's `image_prompt` field.

```python
# Source: CONTEXT.md code_context, get_podcast_style_prompt() default
for scene in all_scenes:
    scene["image_prompt"] = f"{self.style_prompt} {scene['image_prompt']}"
```

Apply after all 3 calls, before writing to disk. This ensures consistent visual style lock per IMG-02 requirements (Phase 6 prep).

### Anti-Patterns to Avoid

- **Do not import Pydantic or use `model_json_schema()` in Podcast.py** — D-01 forbids it. Use the inline dict schema.
- **Do not call Ollama client directly in Podcast.py** — always go through `llm_provider` functions. This is the project's established provider pattern.
- **Do not cache config values** — call `get_podcast_narrator()` and `get_podcast_style_prompt()` each time (project pattern: every getter re-reads config.json on every call).
- **Do not pass `think=True`** to the structured calls — thinking mode adds tokens and slows generation; not needed for creative narration tasks.
- **Do not mutate `self.narrator` mid-method** — build the system prompt from it once and hold it as a local variable.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Structured JSON output | Custom regex extraction from free-text | `generate_text_structured()` with `format=schema` | Ollama enforces schema at generation time; regex fails on nested JSON, encoding edge cases |
| JSON parsing safety | Custom parser or eval() | `json.loads()` with try/except | Standard library handles all edge cases |
| Directory creation | Manual os.path.exists + makedirs guard | `os.makedirs(path, exist_ok=True)` | `exist_ok=True` is atomic and idiomatic |
| Topic slugification | Custom transliteration | `re.sub(r"[^a-z0-9]+", "-", topic.lower())` | Sufficient for ASCII topic strings; project has no Unicode topic requirement |

**Key insight:** The only genuinely new logic is the three-call orchestration loop and the running summary injection. Everything else (LLM calls, JSON parsing, file I/O) reuses existing infrastructure.

## Common Pitfalls

### Pitfall 1: generate_script() Stub Signature Mismatch

**What goes wrong:** The stub is `generate_script(self) -> None`. Adding `topic: str` as a parameter also requires updating the `run()` call site (`self.generate_script()` → `self.generate_script(self.topic)`) and storing topic on the instance.
**Why it happens:** The stub was written before the topic-taking design was finalized.
**How to avoid:** Add `topic: str` to `__init__` and store as `self.topic`; update `run()` to pass it; update `generate_script` signature.
**Warning signs:** `TypeError: generate_script() takes 1 positional argument but 2 were given` at runtime.

### Pitfall 2: Scene Count Mismatch

**What goes wrong:** Ollama may return fewer or more scenes than requested (e.g., 3 instead of 5 for Call 1).
**Why it happens:** LLMs don't always respect count instructions even with schema enforcement.
**How to avoid:** After `json.loads()`, assert `len(scenes) == expected_count`. If wrong count, retry (up to 2 attempts) with a more emphatic prompt like `"YOU MUST RETURN EXACTLY {n} scene objects."`.
**Warning signs:** Final list has != 14 items before writing to disk.

### Pitfall 3: json.loads() on Malformed Output

**What goes wrong:** `generate_text_structured()` returns a string that is not valid JSON (rare with `format=schema`, but can happen with older Ollama versions or very large schemas).
**Why it happens:** Schema enforcement can occasionally fail for complex schemas or under OOM conditions.
**How to avoid:** Wrap `json.loads(raw_json)` in try/except `json.JSONDecodeError`. On failure, retry the call once. If retry fails, raise with context for debugging.
**Warning signs:** `json.JSONDecodeError` at parse time.

### Pitfall 4: Running Summary Consuming Context Budget

**What goes wrong:** The summary for Calls 2+3 grows if concatenated naively, eventually exceeding model context.
**Why it happens:** Injecting "summary of all prior summaries" balloons the prompt size.
**How to avoid:** Only pass the immediately prior act's summary to each call — not a cumulative chain. For Call 3: summarize Calls 1+2 together once (not "summary of summary"). Keep summaries to 3-5 sentences (CONTEXT.md specifies this).
**Warning signs:** Slow inference or truncated output for later acts.

### Pitfall 5: `response["message"]["content"]` Key Access on ChatResponse

**What goes wrong:** `response["message"]["content"]` appears to be dict access, but `ChatResponse` is a Pydantic model with `__getitem__` defined on `SubscriptableBaseModel`. It works but is non-obvious.
**Why it happens:** Ollama SDK uses a custom `SubscriptableBaseModel` that supports both attribute and key access.
**How to avoid:** No action needed — existing pattern in `generate_text_structured()` is correct and tested. Do not change to `response.message.content` style (would also work but inconsistent with codebase).
**Warning signs:** None expected; just understand why it works.

### Pitfall 6: Qwen3 Thinking Tokens in Content

**What goes wrong:** If thinking mode activates, `<think>...</think>` prefixes the JSON output, breaking `json.loads()`.
**Why it happens:** Qwen3 models have a thinking mode; with the Ollama SDK 0.6.1, thinking content is routed to `message.thinking` separately from `message.content`. However, older Ollama server versions (< 0.6) concatenated them.
**How to avoid:** Current project uses Ollama SDK 0.6.1. No action needed at SDK level. As defensive measure, add `/no_think` to the system prompt to explicitly disable thinking mode in Qwen3.
**Warning signs:** `json.JSONDecodeError` on responses that start with `<think>`.

### Full Call Pattern (Act 1: intro + act1)

```python
# Source: YouTube.py generate_hook() + llm_provider.generate_text_structured()
SCENE_SCHEMA = {
    "type": "object",
    "properties": {
        "scenes": {
            "type": "array",
            "items": {
                "type": "object",
                "properties": {
                    "narration": {"type": "string"},
                    "image_prompt": {"type": "string"},
                },
                "required": ["narration", "image_prompt"],
            },
        }
    },
    "required": ["scenes"],
}

narrator = get_podcast_narrator()
system_prompt = (
    f"You are {narrator['name']}, {narrator['persona']}. /no_think "
    "Output ONLY valid JSON matching the provided schema. No markdown."
)

user_prompt = (
    f"Topic: {topic}\n\n"
    "Generate EXACTLY 5 scenes: 1 intro scene + 4 act-1 scenes.\n"
    "intro: hook the viewer, introduce the topic broadly.\n"
    "act1: dive into the first major subtopic with specific facts.\n"
    "Each scene: narration (2-4 spoken sentences), image_prompt (vivid visual description)."
)

raw = generate_text_structured(
    prompt=user_prompt,
    system_prompt=system_prompt,
    schema=SCENE_SCHEMA,
)
call1_scenes = json.loads(raw)["scenes"]  # expect len == 5
```

- `format="json"` mode: Works but produces free-form JSON without field guarantees. Use `format=schema_dict` for reliable field access.

## Open Questions

1. **Topic parameter threading through run()**
   - What we know: `run()` calls `generate_script()` with no arguments currently.
   - What's unclear: Whether topic is supplied at `run()` call time or at `__init__` time.
   - Recommendation: Add `topic: str` to `__init__`, store as `self.topic`, pass `self.topic` to `generate_script()`. Matches YouTube.py pattern (niche/language set at construction).

2. **style_prompt prepend timing**
   - What we know: `get_podcast_style_prompt()` returns a style prefix for `image_prompt`.
   - What's unclear: Whether to ask the LLM to include the style in its output or to postprocess.
   - Recommendation: Postprocess — let the LLM generate a scene-specific visual description, then prepend style prefix after collection. This gives better compositional results than asking the LLM to self-style-prefix.

3. **Retry strategy for scene count mismatch**
   - What we know: LLMs sometimes return wrong scene counts.
   - What's unclear: Whether to retry the whole call or pad/trim.
   - Recommendation: Retry once with a stronger count emphasis. If retry still wrong, raise `ValueError` with count details. Do not pad/trim silently — wrong counts cause downstream phase failures.
