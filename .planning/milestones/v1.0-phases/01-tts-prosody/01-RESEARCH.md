## Summary

Phase 1 is a minimal, surgical change. The existing `Tts.py` instantiates `edge_tts.Communicate(text, voice)` but passes no prosody parameters. Adding `rate=get_tts_rate()` to that one call â€” and adding the corresponding getter to `src/config.py` â€” fully satisfies both TTS-01 and TTS-02.

The installed library (edge-tts 7.2.8, aliased as kittentts in requirements.txt) already accepts `rate` as a keyword-only argument with default `"+0%"`. No library upgrade is needed. The `Communicate.__init__` signature was verified directly against the installed venv package.

No pipeline changes are required outside of `src/classes/Tts.py` and `src/config.py`. The `ffmpeg` mp3â†’wav conversion step and all downstream consumers (Whisper STT, Remotion audio track) are unaffected because they operate on the output WAV file, not the TTS synthesis parameters.

**Primary recommendation:** Add `get_tts_rate()` to `src/config.py` using `.get("tts_rate", "+20%")`, add `"tts_rate": "+20%"` to `config.example.json`, and pass `rate=get_tts_rate()` to `Communicate` in `_edge_tts_synthesize`. Three files, three edits.

### Pattern 1: Config Getter with Safe Default

All config getters in `src/config.py` follow a consistent pattern. The existing getters for optional values use `.get(key, default)` so that a missing key never raises a `KeyError`.

**What:** New `get_tts_rate()` function reads `config.json` and returns the `tts_rate` value with a string default.
**When to use:** Any config value that is optional and has a sensible default.

```python
# Source: src/config.py existing pattern (get_ollama_model, get_whisper_model, etc.)
def get_tts_rate() -> str:
    """
    Gets the TTS speaking rate from config.

    Returns:
        rate (str): edge-tts rate string, e.g. "+20%". Defaults to "+20%" if not set.
    """
    with open(os.path.join(ROOT_DIR, "config.json"), "r") as file:
        return json.load(file).get("tts_rate", "+20%")
```

### Pattern 2: Passing Rate to Communicate

The `_edge_tts_synthesize` async function in `Tts.py` constructs `edge_tts.Communicate`. The `rate` kwarg is added here.

**What:** Pass `rate` as a keyword argument when constructing `Communicate`.
**When to use:** Every TTS synthesis call â€” rate is global, not per-call.

```python
# Source: edge_tts.Communicate.__init__ signature (verified against installed v7.2.8)
async def _edge_tts_synthesize(text: str, output_mp3: str, voice: str, rate: str) -> None:
    import edge_tts
    communicate = edge_tts.Communicate(text, voice, rate=rate)
    await communicate.save(output_mp3)
```

The `TTS.synthesize()` method then fetches the rate from config and passes it down:

```python
# In TTS.synthesize():
from config import ROOT_DIR, get_tts_voice, get_tts_rate

def synthesize(self, text, output_file=...):
    mp3_path = output_file.replace(".wav", ".mp3")
    rate = get_tts_rate()
    asyncio.run(_edge_tts_synthesize(text, mp3_path, self._voice, rate))
    # ... ffmpeg conversion unchanged
```

### Pattern 3: config.example.json Entry

The new config key must appear in `config.example.json` so users know it exists and can tune it.

```json
"tts_rate": "+20%"
```

Place it adjacent to `"tts_voice"` for discoverability.

### Anti-Patterns to Avoid

- **Do not add `pitch` kwarg:** `pitch` is accepted by `Communicate` but silently ignored by Microsoft's server since v6.0.3. Adding it creates false expectations. Confirmed by PITFALLS.md Pitfall 3.
- **Do not pass `rate` as a bare integer:** `rate=20` is wrong. Must be a string with explicit sign and percent: `rate="+20%"`. The `Communicate` constructor accepts only string format.
- **Do not store rate on `self` in `TTS.__init__`:** The existing pattern in this codebase re-reads `config.json` on every call (no caching). Fetching rate inside `synthesize()` (not `__init__`) keeps the class consistent with that convention. This allows rate changes to take effect without restarting the process.
- **Do not hard-code `"+20%"` in the `Communicate` call:** The whole point of TTS-02 is that rate is tunable from config without code changes.

## Common Pitfalls

### Pitfall 1: Pitch Parameter Silently Ignored

**What goes wrong:** Developer adds `pitch="+50Hz"` expecting vocal variety. Audio sounds identical to the no-pitch call. No error is raised.

**Why it happens:** Microsoft's TTS server removed pitch support on their backend. edge-tts v6.0.3 removed the CLI `--pitch` option for this reason. The Python `pitch` kwarg still exists in the constructor signature but produces no effect.

**How to avoid:** Do not add `pitch` to the `Communicate` call. Do not expose a `tts_pitch` config key. Document in code comments that pitch is a no-op.

**Warning signs:** Generated audio sounds identical regardless of pitch value.

---

### Pitfall 2: Rate String Format Wrong

**What goes wrong:** Passing `rate=20` (integer) or `rate="20"` (no sign) to `Communicate` causes a runtime error or silently uses the default rate.

**Why it happens:** The API requires a percentage string with an explicit sign character: `"+20%"` or `"-10%"`. A bare number or unsigned string is not the expected format.

**How to avoid:** Always format the rate string with sign and `%`. The default in `get_tts_rate()` is already `"+20%"` â€” users editing `config.json` must include the sign.

**Warning signs:** Audio sounds unchanged (default rate) even when config is set; or a `ValueError` from the aiohttp layer at synthesis time.

---

### Pitfall 3: Rate Fetched at `__init__` Instead of `synthesize`

**What goes wrong:** If `get_tts_rate()` is called in `TTS.__init__` and stored as `self._rate`, then a config change between runs (e.g., by a cron job editing config.json mid-session) won't take effect until the process is restarted. This diverges from the project's established convention of re-reading config on every call.

**Why it happens:** Looks like a natural optimisation. The existing `self._voice = get_tts_voice()` in `__init__` appears to set the precedent â€” but this is a pre-existing inconsistency, not the intended pattern.

**How to avoid:** Read `get_tts_rate()` inside `synthesize()`, consistent with how all other config values are read throughout the codebase (fresh read on every invocation).

**Warning signs:** Rate changes in config.json don't take effect until process restart.

---

### Pitfall 4: Missing Key Causes KeyError Instead of Graceful Default

**What goes wrong:** Using `json.load(file)["tts_rate"]` (bracket access) instead of `.get("tts_rate", "+20%")` means any user whose `config.json` predates this feature will get an unhandled `KeyError` at runtime.

**Why it happens:** Direct key access is the pattern for required config values. `tts_rate` is optional â€” it should have a default.

**How to avoid:** Always use `.get(key, default)` for optional config values, matching the pattern in `get_ollama_model`, `get_whisper_model`, `get_stt_provider`, etc.

**Warning signs:** `KeyError: 'tts_rate'` in the TTS stage for users who haven't updated `config.json`.

## Open Questions

1. **Rate format validation**
   - What we know: `Communicate` accepts any string for `rate`; malformed values (e.g., `"20"` without sign) may silently use the default or raise at the aiohttp level
   - What's unclear: Whether edge-tts validates the format client-side before sending to Microsoft, or whether a malformed string silently falls back to default rate
   - Recommendation: Document valid format in the config.example.json comment; the planner may optionally add a lightweight format guard in `get_tts_rate()` (`if not re.match(r'^[+-]\d+%$', rate): return "+20%"`)
